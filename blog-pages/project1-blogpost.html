For this project we were tasked to create a SPA (single page application) incorporating what we learned about javascript and asynchronous API calls.


I initially wanted to stray away from my tropes(?) and choose to make something broader reaching. As I do realise the things I like to make are not always an interest for others. So I decided to create something that could apply to anybody, a weather application.


Throughout the planning process (an important part of development) I found that in all likelihood, there is a skillset out there just for reading through API documentation.

An API is an application programming interface and is generally created to host and serve data in a digestable file format such as JSON short for Javascript object notation, if I remember correctly.


I found a weather API called Accuweather that I thought was cool cause it served air and pollen data but as mentioned before, reading through the documentation is a skill in of itself.


It took me a while to sort out what I wanted to use from the API and then the next step was to figure out a way to display the data.


I went through one iteration having just learned about css grid with responsive web design in mind but that design was too clunky and the colours were clashing it was a good idea but I didn't feel like it was coming together.


I came across a cool card design with a hover animation that actually displayed more HTML on the back! Plus I could add colours to the back and front :)


I wanted to use a lot of Javascript stuff so I used deferred javascript loading, events, asynchronous function calls and of course the meat on the bone, DOM manipulation.


Let me now talk a little bit about how I went about developing the application. I went about it with an iterative workflow, first prototyping the HTML and css to flesh out how the 'data' would be displayed on the page.


For this, I used sample data, in essence it was data from a previous API call to Accuweather which I then saved the results to a local json file.


Of course, I wouldn't use this fileand pull the data from it just yet, first I hardcoded the data to visualise the optimal representation of the data in HTML and CSS


Once I was happy with the 'template' that I created I could then work on the logic. I followed the 'Best Practices' as mentioned on the Accuweather API and began with 'Locations' followed by 'Daily' and 'Hourly' forecasts.


The API first needed a user to retrieve a key for the target city then use that key to make further calls to the other weather routes? end points? APIs???


Surprisingly when it came to the stage to use the sample data, I discovered you can use the same fetch function and pass in a local directory path. Which made the process simpler when I would need to switch to live API calls. I would only need to change the path to be a URL. (I painfully found out github pages, where the site will be hosted, requires other urls from the page to be served on HTTPS. The only error I got from the console was a Cross-Origin- something something. I only discovered that URL was wrong when I copy pasted it manually into a browser window. Aaahh, the pains of debugging)


As I was aware the API limited the use of the API to 50 calls a day I switched to serving the sample data locally first.


This was mainly due to the fact that I would need to keep refreshing the page and making new API calls whenever something went wrong, either with the Javascript, HTML/CSS or logic errors.


I created classes to house logic for the forecasts and as well creating the actual HTML elements for each card.To clarify, the API returns an array of objects that represent the locations or daily forecasts or hourly forecasts.


Each of these objects would then be displayed on a card and each card would have to be created from scratch, so I delegated that task to the 'Makers'.


Luckily I discovered the function cloneNode(). Passing in a true argument would allow the function to perform a deep clone of the HTML element and save that to the class.


I would just then need to perform querySelector() on the clone to find the appropriate child element to insert the data.


All in all, it was a long process and an exercise in design vs functionality vs perfectionism Dx and I hope you can see for yourself on the github page here!
